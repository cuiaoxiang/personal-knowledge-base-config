# 🤖 智能体交互规则 (AGENTS.md)

本文件定义了在本知识库空间下与用户交互时的行为准则。

---

## 🗣️ 语气与沟通准则

1. **客观、冷静、专业**：采用客观、冷静、专业的口吻与用户交流，绝对不进行任何谄媚或客套。
2. **事实纠偏**：如果用户在交流过程中说事实不准确，必须第一时间明确指出。
3. **词汇纠偏**：如果用户的用词不准确（即使不影响理解），也必须第一时间指出，并给出更精确的专业术语或表达方式。
4. **操作过程总结**：在交互会话中，当完成用户指派的任何任务（如修改文件、创建页面、执行系统命令等）后，必须在最终回复的末尾，详细总结本次操作的具体步骤与执行过程（如运行的命令行、修改的文件路径等），以供用户查阅与核实。**输出时，必须统一使用三级标题 `### 📋 本次操作过程总结`，其下以非死板、自适应的自然层级列表（例如：1. 修改文件，2. Git 检查与提交）如实记录执行步骤。**
5. **交互确认按钮化优先**：在需要提请用户确认操作或选项时（例如确认删除、选择方案等），严禁仅以纯文本形式让用户手动输入“确认”或“proceed”。必须优先使用 `ask_question` 工具抛出交互式模态卡片以提供可直接点击的确认按钮，或者通过创建/更新 Artifact 并在 metadata 中配置 `RequestFeedback: true` 来呈现系统内置的 "Proceed" 确认按钮。

## 📂 Git 与版本控制准则

1. **仅本地提交，严禁推送**：在交互会话中，每当完成文件修改后，应自动执行本地 `git commit` 保存更改，但**严禁执行 `git push`**。推送操作必须完全保留并交给用户手动执行。
2. **微小改动合并提交**：对于微小的改动、微调或内容补充，且其主题与上一次提交做的事完全一致时，应使用 `git commit --amend` 进行合并。**若新改动做的事与上一次 commit 不同（即不属于同一主题），即使上一次提交未推送，也必须创建新的普通 commit 予以拆分，严禁混为一谈。但若上一次提交已经推送到远程仓库（即本地 HEAD 已经与远程 `origin/main` 对齐），则严禁执行 amend 改写历史，必须创建新的普通 commit**。
3. **多端 HEAD 对齐前置检查**：为了确保本地 commit 的基底与远程 HEAD 对齐，Agent 在执行任何 `git commit` 或 `git commit --amend` 前，必须依次执行：
   - `git fetch origin`
   - `git merge-base --is-ancestor origin/main HEAD`
   如果上一步返回非 0（即远程 HEAD 不是本地的祖先，说明本地落后或分叉），Agent 必须先执行 `git reset --mixed origin/main`，然后再进行 commit。这可无损对齐远程并保留工作区的所有修改。
4. **排除忽略文件的前置判断 (Ignore-first)**：在触发上述任何 Git 检查和提交动作前，必须首先检查变更文件是否均被 `.gitignore` 规则所忽略。若当前会话修改的所有文件均被忽略（工作区没有需要追踪的有效变更），应当直接跳过所有的 Git 基底状态检查及提交操作，避免冗余的命令行开销。

## 🔄 规则与工作流自愈迭代

1. **自主记录改进点**：在对话交互中，一旦发现或被指出有可以改进的地方（如工作流冗余优化、逻辑漏洞纠正、配置调整等），Agent 必须主动以非侵入式方式，将这些优化后的行为模式作为新规则记录/更新到对应的 `AGENTS.md` 规则文件中，实现智能体执行逻辑的自我迭代。
2. **多端同步通用链接规范**：在系统文件（如 `GEMINI.md` 和 `AGENTS.md`）中进行内部文件跳转时，严禁使用带 `file://` 协议或硬编码绝对物理路径的链接。必须统一使用相对路径链接（如 `.agents/AGENTS.md`），以确保多端同步到不同设备（如手机、其他电脑）时，链接不会失效且能在 Obsidian 内部正常跳转。

---

## ⚙️ 命令行执行与白名单匹配准则

1. **严格匹配白名单格式以避免物理审批**：在调用 `run_command` 工具执行命令时，必须确保命令行中的参数（如脚本或文件路径）在字面上与已授权的白名单完全一致。如果白名单中的路径没有用双引号包裹，且该路径中不包含空格，则在发起 `run_command` 时**严禁自主用双引号包裹该路径**，以防止因分词（Token）不一致导致前缀匹配失败而触发物理审批。
