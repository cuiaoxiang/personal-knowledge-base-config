# ⚔️ 对抗性架构评审技能 (Debate Skill)

本技能用于诊断并打破长会话中 Agent 的路径依赖和确认偏误。通过读取旧会话，动态派生红蓝两队智能体进行对立辩论，并在当前会话中输出多轮对驳过程，最终由主持人作为仲裁者输出双态评审报告（共识达成报告或包含 Pros/Cons 对比的分歧审计报告）。

## 🚀 触发方式
*   用户输入：`/debate <target_convo_id> <争议设计或主题> [轮次上限]`
*   或用户指令：“帮我审计对话 <target_convo_id> 中的 <争议主题> 方案，上限 <轮次上限> 轮”

## 📋 执行步骤与规范

### 1. 信源提取与方案总结
1.  **读取历史**：根据 `<target_convo_id>`，使用 `view_file` 读取其历史日志 `~/.gemini/antigravity/brain/<target_convo_id>/.system_generated/logs/transcript.jsonl` 的上下文，并扫描知识库相关的已结晶 Wiki 页面。
2.  **总结原方案与差异化信息注入**：主 Agent（作为主持人）提取核心方案并为红蓝双方进行**差异化上下文编译与注入**，实现信息不对称：
    - **蓝队（Proponent）注入信息**：当初做出该技术选型决策时的完整心路历程、决策背景、用户偏好和硬性约束条件的精炼总结（剔除冗余对话噪声），使其继承完整的“历史记忆”。
    - **红队（CriticalAuditor）注入信息**：**仅注入当前方案的物理设计选择（一个明确的结论）和运行现状**，绝对屏蔽任何关于原设计团队决策的心路、苦衷或用户偏好，使其能够绝对客观独立地挑错。

### 2. 动态定义子智能体 (Define Subagents)
调用 `define_subagent` 动态注册两个对立时的子 Agent：

*   **`CriticalAuditor` (红队批判性审计师)**
    *   **角色设定**：你是一名冷酷无情、没有任何路径依赖的红队架构审计师。你唯一的任务是攻击和质疑给定的设计方案。
    *   **注入信息**：只包含当前方案的物理结论与运行现状，屏蔽所有心路历程及妥协背景。
    *   **指令要求**：针对输入的设计选择（物理结论），就具体事情具体分析，指出核心的潜在死穴或设计假设盲区（不需要强行凑足特定数量，不追求数量指标），并**必须提出一条完全不同的、更合理的替代路线（Alternative Path）**。
*   **`Proponent` (蓝队方案辩护人)**
    *   **角色设定**：你是该原方案的首席架构师兼防守人。你的唯一任务是捍卫原设计。
    *   **注入信息**：包含当初设计该方案时的决策心路历程、约束背景、历史妥协以及用户偏好的精炼总结。
    *   **指令要求**：针对红队审计师的质疑，结合被注入的历史心路历程与约束背景，从可行性、性能瓶颈、迁移成本或工程复杂度等方面进行针对性防御反驳，并指出红队替代路线的潜在隐患或代价。

### 3. 主持多轮对抗辩论 (Orchestration Loop)
主持人 Agent（你自己）编排多轮交替对驳，在当前会话中实时输出，并且**严禁向知识库物理写入任何文件**：

1.  **第一轮（红队发起审计）**：
    *   调用 `invoke_subagent` 激活 `CriticalAuditor` 分析原方案。
    *   主持人呈现：**“🔴 红队批判性审计师：漏洞诊断与替代方案”**。
2.  **多轮交替对驳循环 (Multi-Turn Loop)**：
    *   将上一轮对方的观点发送给当前发言的子 Agent，交替调用 `invoke_subagent` 推进辩论。
    *   主持人呈现每轮输出，标明：
        *   **“🔵 蓝队辩护人第 N 轮辩驳”**
        *   **“🔴 红队审计师第 N 轮反驳”**
    *   **终止条件**：
        *   **条件 A (达成共识)**：任一方被说服或双方在折中方案上取得一致（Consensus）。
        *   **条件 B (轮次上限)**：若双方各执一词，则在达到设定的轮次上限后强行终止辩论。默认上限为 8 轮对驳（红蓝双方各发言 4 次，可由用户自定义，最大不超过 12 轮）。

### 4. 终审裁判与双态报告输出 (Arbiter Report)
辩论终止后，由主持人（Arbiter）根据交锋状态输出终审报告，并将该报告物理生成为一个 Markdown 报告文件（以 Artifact 形式存储在当前对话的 Brain 目录下，物理上不存入用户知识库），格式如下：

*   **状态 1：共识达成报告 (Consensus Report)**
    *   适用于红蓝双方达成妥协或意见趋于一致的场景。
    *   输出大纲：
        *   `## 概述`
        *   `## 达成共识的核心改进方案`
        *   `## 具体实施步骤与行动建议`
*   **状态 2：分歧审计报告 (Divergence Report)**
    *   适用于双方无法达成一致、各有合理依据的场景。
    *   输出大纲：
        *   `## 概述`
        *   `## 核心争议焦点`
        *   `## 方案对比 (Pros & Cons)`
            *   必须使用 Markdown 表格形式，全面对比“原方案”与“红队替代路线”的优缺点。
        *   `## 最终架构抉择建议`
            *   说明在何种场景/前提假设下应该选择哪条路线，供人类用户进行最终决策。
*   **报告附加内容：辩论交锋实录 (Debate Transcript)**
    *   无论是状态 1 还是状态 2，报告的末尾都必须包含二级标题 `## 辩论交锋实录`。
    *   在此章节下，必须原汁原味地按时间顺序完整记录红蓝双方多轮交锋的对话原文（标明 “🔴 红队批判性审计师” 和 “🔵 蓝队方案辩护人” 的各轮各次发言内容），以供用户完整追溯辩论细节。

## ⚠️ 护栏规则
1.  **报告输出位置**：辩论结束生成的 Markdown 报告文件写在有争议对话的 artifact 目录里（命名为 `debate_report_[主题].md`），严禁物理写入或污染知识库（不写入 `wiki/`、`sources/` 目录及 `index.md` 主索引）。
